# 工业视觉测试用例 Agent 工作台设计文档

> 目标：将现有基于静态 Markdown 文档的工业视觉测试用例生成 Agent，升级为一个基于 Python 的模块化 Agent 工作台。第一版采用 Web 优先、桌面可封装的路线，支持从零生成测试用例、优化已有用例、知识库驱动生成、覆盖分析与多格式导出。

## 1. 背景与现状

当前项目已经具备以下背景资产：

- `skill.md`：定义工业视觉测试用例生成的能力范围
- `rules.md`：定义生成规则、交互规则和输出规则
- `prompt-templates.md`：定义系统指令模板、用户输入模板和输出模板
- `knowledge-base.md`：提供工业视觉软件测试知识库

这些内容已经覆盖了工业视觉测试用例生成所需的核心知识，但它们目前仍然是静态文档，存在以下限制：

1. 不能直接作为可运行系统使用
2. 不能通过界面完成输入、生成、优化和导出
3. 不能对知识条目进行结构化维护和检索
4. 不能形成清晰的 Agent 协作流水线
5. 不能为后续桌面化和模型接入提供稳定基础

因此需要将当前项目升级为一个真正可运行的 Python 系统。

## 2. 项目目标

升级后的第一版系统应满足以下目标：

1. 提供 Python 驱动的交互界面
2. 支持从零生成工业视觉测试用例
3. 支持导入已有测试用例并进行优化
4. 支持知识库驱动的测试点补全和风险识别
5. 支持覆盖分析和缺失项提示
6. 支持 Markdown、XMind 风格文本、JSON 导出
7. 采用 Web 优先、桌面可封装的实现路线

## 3. 设计原则

### 3.1 架构原则
- 采用单体应用 + 内部模块化 Agent 流水线
- 对外提供统一应用，对内拆分为多个职责清晰的 Agent 模块
- 不在第一版引入分布式多 Agent 调度

### 3.2 技术原则
- 以 Python 为核心
- 以 Web 为第一界面形态
- 预留桌面封装能力
- 第一版优先采用本地结构化文件作为知识存储
- 第一版优先采用规则驱动 + 模板驱动，后续可插拔接入 LLM

### 3.3 范围原则
- 第一版聚焦可运行、可维护、可扩展
- 不过度设计权限系统、数据库平台和企业级集成
- 不实现真正的 `.xmind` 二进制导出，先提供 XMind 风格树形文本导出

## 4. 总体架构

系统采用“单体应用 + 内部模块化 Agent 流水线”的方式实现。

### 4.1 架构形态
- 外部表现为一个 Python 应用
- 内部由多个功能模块组成，每个模块承担一个 Agent 角色
- 所有模块共享统一的数据模型和知识库

### 4.2 模块划分

#### 4.2.1 需求解析模块
职责：
- 解析用户输入
- 识别功能描述、参数配置、流程步骤、测试深度、输出格式
- 区分当前任务模式：从零生成 / 优化已有用例
- 生成结构化任务对象

#### 4.2.2 知识库管理模块
职责：
- 读取和维护知识条目
- 管理工具知识、测试策略、常见缺陷经验、输出模板知识
- 为生成链路和优化链路提供检索支持

#### 4.2.3 测试点规划模块
职责：
- 从需求和知识库中识别测试点
- 输出功能测试点、参数测试点、边界值测试点、异常场景点、流程测试点
- 为后续测试用例生成提供中间结果

#### 4.2.4 测试用例生成模块
职责：
- 将测试点转换为标准化测试用例
- 统一输出测试用例 ID、名称、优先级、严重程度、前置条件、步骤、预期结果等

#### 4.2.5 覆盖分析模块
职责：
- 分析已覆盖的测试类型
- 识别缺失测试项
- 提示参数、流程、异常、工具交互等方面的缺口
- 生成覆盖分析摘要和风险提醒

#### 4.2.6 用例优化模块
职责：
- 解析已有测试用例
- 识别弱覆盖项、模糊步骤、缺失预期结果
- 自动补充测试点并输出增强版测试用例

#### 4.2.7 导出模块
职责：
- 将统一内部模型导出为 Markdown、XMind 风格文本、JSON
- 支持页面预览和下载

## 5. 页面结构设计

系统第一版采用左侧导航 + 中央工作区 + 右侧结果面板的三栏工作台布局。

### 5.1 左侧导航
用于切换核心功能区：
- 工作台首页
- 从零生成
- 用例优化
- 知识库管理
- 模板中心
- 导出记录

### 5.2 中央工作区
作为主操作区域，承载输入表单、知识编辑、模板编辑等核心内容。

### 5.3 右侧结果面板
用于实时显示：
- 当前生成摘要
- 测试点规划结果
- 覆盖分析
- 风险提醒
- 可导出格式预览

### 5.4 工作台首页
首页作为总控台，包含：
- 快速开始：新建生成任务、导入用例优化任务
- 最近任务：显示最近执行记录
- 知识库状态：显示知识条目数、工具类型数、更新时间
- 系统建议：提示知识不足、模板缺失、覆盖不足等信息

### 5.5 从零生成页
包含以下输入区域：
- 基础信息：任务名称、工具名称、测试深度、输出格式、优先测试类型
- 功能描述：输入功能说明、预期效果、业务目标
- 参数配置：表格形式填写参数名、说明、取值范围、默认值、是否关键参数
- 流程步骤：逐条输入业务流程步骤
- 附加约束：环境要求、硬件型号、测试数据、兼容要求、性能要求

页面底部操作：
- 生成测试点
- 生成完整测试用例
- 清空输入

### 5.6 用例优化页
包含以下区域：
- 原始用例输入：支持粘贴 Markdown、表格文本或普通文本
- 优化目标选择：边界值补全、异常场景补全、步骤增强、预期结果增强、去重合并、覆盖率提升
- 参考上下文：工具名称、功能说明、参数列表、相关知识条目
- 优化输出：优化建议摘要、缺失项清单、优化后完整用例、修改对比视图

### 5.7 知识库管理页
布局：左侧分类树 + 右侧知识编辑区

支持分类：
- 软件架构
- 工具知识
- 通用测试方法
- 历史缺陷经验
- 参数规则
- 模板规则

知识条目支持字段：
- 标题
- 分类
- 适用模块
- 内容
- 测试重点
- 常见问题
- 标签
- 启用状态

### 5.8 模板中心页
用于管理：
- 系统初始化模板
- 工具测试模板
- 参数测试模板
- 流程测试模板
- 优化模板
- 导出模板

支持操作：
- 查看
- 编辑
- 复制为新模板
- 设置默认模板
- 预览变量占位

### 5.9 结果面板 Tab
统一包含：
- 预览
- 测试点
- 覆盖分析
- 导出

### 5.10 视觉风格
建议风格：
- 深色主题优先
- 工业感、工作站风格
- 明确的卡片层级
- 高对比强调按钮
- 状态标签和覆盖率标识

## 6. 技术选型

### 6.1 推荐技术栈
- 后端：FastAPI
- 页面模板：Jinja2
- 局部交互：HTMX
- 样式：Tailwind CSS
- 页面增强：少量原生 JavaScript
- 数据存储：本地 YAML / JSON / Markdown
- 桌面封装预留：PyWebview

### 6.2 选型理由

#### FastAPI
- 适合 Python 主导的应用
- 支持路由、文件上传、异步接口
- 便于后续增加 API 能力和模型调用

#### Jinja2 + HTMX
- 页面开发速度快
- 适合工作台类应用
- 可以实现局部刷新，无需一开始做完整前后端分离

#### Tailwind CSS
- 适合快速构建三栏工作台布局
- 有利于统一页面风格和状态组件样式

#### 本地文件存储
- 便于快速落地
- 便于人工维护知识库和模板
- 便于版本管理
- 后续可平滑迁移到 SQLite 或其他存储方案

#### PyWebview
- 可以在第二阶段将本地 Web 应用封装为桌面程序
- 无需重写界面逻辑

### 6.3 不作为第一版主方案的技术
- 不采用 Streamlit 作为正式工作台框架
- 不采用前后端完全分离架构
- 不引入复杂数据库平台
- 不引入 Electron/Tauri 作为第一版桌面方案

## 8. Agent 工作流设计

### 8.1 Agent 角色

#### 8.1.1 需求解析 Agent
输入：功能描述、参数表、流程步骤、测试目标、已有用例文本
输出：结构化任务对象

#### 8.1.2 知识检索 Agent
输入：工具名称、模块名称、参数名、流程步骤、优化目标
输出：相关工具知识、测试策略、风险点、模板

#### 8.1.3 测试点规划 Agent
输出：
- 功能测试点
- 参数测试点
- 边界值测试点
- 异常场景点
- 流程测试点
- 性能/稳定性候选点

#### 8.1.4 测试用例生成 Agent
将测试点转换为标准测试用例。

#### 8.1.5 覆盖分析 Agent
分析缺失项、风险项和覆盖度。

#### 8.1.6 用例优化 Agent
针对已有用例做增强和补全。

### 8.2 主流程

#### 流程 A：从零生成
```text
用户输入需求
→ 需求解析 Agent
→ 知识检索 Agent
→ 测试点规划 Agent
→ 测试用例生成 Agent
→ 覆盖分析 Agent
→ 导出模块
```

#### 流程 B：优化已有用例
```text
用户导入已有用例
→ 需求解析 Agent
→ 知识检索 Agent
→ 覆盖分析 Agent
→ 用例优化 Agent
→ 导出模块
```

### 8.3 第一版智能实现方式
采用三层组合：

#### 规则驱动
- 参数范围存在时自动生成边界值测试点
- 流程步骤较多时自动生成流程完整性测试点
- 工具知识中的风险项自动转化为异常测试候选

#### 模板驱动
- 功能测试模板
- 参数测试模板
- 流程测试模板
- 优化建议模板

#### 可插拔 LLM 接口
后续可接入在线或本地模型，但第一版不依赖它作为基础能力。

## 9. 输出格式设计

### 9.1 统一内部模型
系统内部统一采用结构化测试用例模型，例如：

```python
{
  "case_id": "VIS-AREA-FUNC-001",
  "name": "验证面积检测工具在标准矩形ROI下的基本检测功能",
  "test_type": "功能测试",
  "priority": "高",
  "severity": "严重",
  "preconditions": ["已创建物料方案", "已导入标准参考图像"],

  "expected_results": ["成功输出面积值", "结果状态正确显示为合格"],
  "remarks": "适用于标准参考图像场景"
}
```

### 9.2 Markdown 导出
支持：
- 单条测试用例
- 测试用例集
- 覆盖分析报告

### 9.3 XMind 风格文本导出
第一版输出树形文本结构，用于贴近用户现有工作习惯。

### 9.4 JSON 导出
用于系统内部保存、二次开发和后续集成。

### 9.5 页面预览视图
右侧结果区支持：
- 测试点视图

## 12. 最终结论

本项目第一版应升级为一个基于 Python 的模块化 Agent 工作台：

- 形态上：Web 优先，桌面可封装
- 架构上：单体应用 + 多阶段 Agent 流水线
- 能力上：从零生成 + 用例优化 + 知识库驱动 + 覆盖分析
- 技术上：FastAPI + Jinja2 + HTMX + Tailwind CSS
- 存储上：本地结构化知识库
- 扩展上：预留 LLM 接入和桌面封装能力

该方案能够在第一版实现可落地、可维护、可扩展的目标，同时保留后续继续升级为更强 Agent 系统的空间。
